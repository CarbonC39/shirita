# i18n Plan 2 — String Extraction (Views)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Extract every hard-coded English UI string out of the six view components into the four i18n catalogs, keeping the parity test and `vue-tsc` green throughout.

**Architecture:** One task per view. Each task follows the same repeatable extraction procedure: find the strings, add keys to `en.ts` (source of truth), translate the same keys into `zh-Hans`/`zh-Hant`/`ja`, rewrite the template/script to call `$t`/`t`, then let the parity test + `vue-tsc` catch any drift. Builds directly on Plan 1's infrastructure.

**Tech Stack:** Vue 3 `<script setup>`, vue-i18n@^10, Vitest, vue-tsc.

**Spec:** `docs/superpowers/specs/2026-06-16-i18n-zh-ja-design.md` (§5, §6, §8).

**Prerequisite:** Plan 1 complete (i18n instance, catalogs, parity test, global test injection all in place).

## The Repeatable Extraction Procedure

Apply these steps in **every** task below. They are written once here; each task names its view, lists its strings, and points back to this procedure.

**A. Find the strings.** In the target `.vue` file, list every user-visible English string:
- Template text nodes: `>Loading…<`, `>No chats yet<`, button labels.
- Bound attributes that are user-visible: `placeholder="..."`, `title="..."`, `aria-label="..."`.
- Script-produced strings assigned to anything rendered: `error.value = 'Failed to …'`, ternaries like `status === 'testing' ? 'Testing…' : 'Test connection'`.

Find candidates with (from repo root):
```bash
# placeholder/title/aria attributes
grep -nE '(placeholder|title|aria-label)="[A-Za-z]' shirita-ui/src/views/<View>.vue
# template text and string literals worth reviewing by eye
grep -nE '>[A-Za-z][^<{]*<|'\''[A-Z][a-z]' shirita-ui/src/views/<View>.vue
```
Grep is a starting point, not the authority — read the file and decide per string.

**Do NOT extract** (spec §6): user content (definition names/content, message text, template names, variable names); system identifiers (`def_type` ids, `owner_kind`, `role` strings, provider source keys like `openai`/`anthropic`); `data-test` values; CSS classes; the `/* custom CSS */` code placeholder and other non-prose code.

**B. Add keys to `en.ts`.** Add a nested object under the view's namespace. Use descriptive camelCase keys. For any "number + countable noun" string, use the vue-i18n plural form (en: `'… {count} thing | … {count} things'`; zh/ja: single form). Example shape:
```ts
  home: {
    empty: 'No chats yet',
    newChat: 'New chat',
  },
```

**C. Translate the same keys** into `zh-Hans.ts`, `zh-Hant.ts`, `ja.ts` — identical key paths, localized values. Traditional Chinese uses Taiwan/HK vocabulary (軟體/設定/匯入/影片), not character-mapped Simplified. Plural strings stay single-form in zh/ja.

**D. Rewrite the component:**
- Template text → `{{ $t('home.empty') }}`.
- Attribute → `:placeholder="$t('home.searchPlaceholder')"` (note the leading colon — it becomes a binding).
- Script: add `import { useI18n } from 'vue-i18n'` and `const { t } = useI18n()` in `<script setup>`, then `error.value = t('home.loadFailed')`. For counts: `t('common.tokensEstimate', count)`.

**E. Verify** (run after each view; commands identical every task):
```bash
npm --prefix shirita-ui run test -- src/locales/parity.test.ts   # key parity
npm --prefix shirita-ui run build                        # typed keys + no missing
npm --prefix shirita-ui run test --                              # nothing regressed
```
The parity test fails loudly if any locale is missing a key you added to en; `vue-tsc` fails on a typo'd `$t` key. If an existing component test asserted the old English literal and that string now comes from `en` (still English by default in jsdom), it keeps passing unchanged.

**F. Commit** with `feat(ui): i18n <View> strings` and the standard trailer.

### Worked Example — extracting HomeView (Task 1 in full)

This is the template for Tasks 2–6; they are abbreviated and reference back here.

Suppose `HomeView.vue` contains:
```html
<p v-if="loading">Loading…</p>
<p v-else-if="!chats.length">No chats yet</p>
<input placeholder="Search chats" />
<button>New chat</button>
```

**B.** Add to `en.ts` (inside the top-level object, keeping namespaces sorted sensibly):
```ts
  home: {
    loading: 'Loading…',
    empty: 'No chats yet',
    searchPlaceholder: 'Search chats',
    newChat: 'New chat',
  },
```
> If `loading` / `searchPlaceholder` are generic and reused elsewhere, prefer `common.loading`. For the worked example assume HomeView-specific.

**C.** Add the same `home` block to each translation:
```ts
// zh-Hans.ts
  home: { loading: '加载中…', empty: '暂无对话', searchPlaceholder: '搜索对话', newChat: '新建对话' },
// zh-Hant.ts
  home: { loading: '載入中…', empty: '尚無對話', searchPlaceholder: '搜尋對話', newChat: '新增對話' },
// ja.ts
  home: { loading: '読み込み中…', empty: 'チャットがありません', searchPlaceholder: 'チャットを検索', newChat: '新規チャット' },
```

**D.** Rewrite the template:
```html
<p v-if="loading">{{ $t('home.loading') }}</p>
<p v-else-if="!chats.length">{{ $t('home.empty') }}</p>
<input :placeholder="$t('home.searchPlaceholder')" />
<button>{{ $t('home.newChat') }}</button>
```

**E.** Run the three verification commands → all green.

**F.** Commit.
